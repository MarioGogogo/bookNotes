# 单页面应用的代码分割

# 解决什么问题?

有效解决传统单页面应用脚本过大,加载时间过长的问题.

首先回顾下不采用代码分割的 SPA 方式

从生产--->上线阶段.所有应用打包成 bundle.js.在不采用服务端渲染的条件下.打包脚本包含所有体积很大.如果增加第三方依赖包.体积问题可想而知.

让我们思考整个过程带来的问题:

任何脚本的改动.重新生成一个 bundle.js.作为用户每次 bundle.js 发生变化重新下载..

如果我们想到用"缓存".包住一些稳定的脚本.比如依赖库.我们将 bundle.js 分割成 vendor.js 和 app.js.

包名 说明
vendor.js 所有稳定脚本打包进这里
app.js 业务代码

# 基于业务的代码分割

进一步思考:对 app.js 脚本是否可以再分割?

比如应用包含,首页,登录页,和设置页三个页面.进入不同的页面时候.所加载的脚本不同.

# 合理选择分割纬度

三种分割方式

  1. 按业务逻辑和依赖 (前面已经介绍过了)

  2. 按路由分割 需要整个路由的不同页面彼此独立.react-router 对实现路由层面的代码分割比较支持. 但是这种分割方式会带来代码沉余.比如 20 多个路由.分割出 20 多个页面.20 多个脚本很容易产生重复的代码

  3. 按组件分割 组件分"UI 组件"和"容器组件".代码分割时候.由容器组件控制,在容器组件实现按需加载 UI 组件.根据条件是否加载合适的 Ui 组件

# 合理的加载时机

分为 消极加载 和 积极加载

消极加载: 指不需要用户额外的交互便进行加载,加载由组件的某个生命周期函数或者相关的某些行为触发. 触发不依赖用户动作.

积极加载: 则依赖用户动作和页面进行交互.比如点击鼠标才触发 LoadableNewsPage 的 preload 事件
